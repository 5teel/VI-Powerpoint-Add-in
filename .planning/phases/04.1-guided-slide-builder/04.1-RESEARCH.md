# Phase 04.1: Guided Slide Builder - Research

**Researched:** 2026-03-24
**Domain:** React wizard UI, client-side image processing, Office.js ShapeFill image insertion
**Confidence:** HIGH

## Summary

Phase 4.1 adds a "Build Slide" tab containing a 4-step wizard (Brand -> Image -> Purpose -> Review) that guides users through creating branded, image-inclusive slides. The wizard reuses the existing pipeline (`streamCubeAI` -> `extractSlideContent` -> `insertSlide`) with a new `buildGuidedPrompt()` function that incorporates brand and purpose context. The image upload uses standard browser FileReader + canvas APIs for client-side resize, stored as base64 in React state, and inserted via the existing `ShapeFill.setImage()` pattern already proven in `placeholderRenderer.ts`.

Fluent UI v9 does NOT have a native wizard/stepper component. The wizard must be built from existing Fluent UI primitives (Button, Input, Text, Card, ProgressBar). This is straightforward -- the wizard is a simple linear state machine with 4 steps.

**Primary recommendation:** Build a custom `WizardPanel` component with step state managed via `useState<1|2|3|4>`, using Fluent UI v9 primitives for each step. Extend `insertSlide()` to accept an optional `productImageBase64` parameter and render it via `ShapeFill.setImage()` on a geometric shape in an image region.

### Pattern 1: Wizard State Machine
**What:** Single parent component manages step number and accumulated wizard data in state. Each step is a pure presentational component receiving data + callbacks.
**When to use:** Linear wizards with < 8 steps and no branching logic.
**Example:**
```typescript
// WizardPanel.tsx
interface WizardData {
  brandName: string;
  productImageBase64: string | null;
  purpose: string;
}

type WizardStep = 1 | 2 | 3 | 4;
type BuildState = "idle" | "building" | "built" | "failed";

const WizardPanel: React.FC = () => {
  const [step, setStep] = useState<WizardStep>(1);
  const [data, setData] = useState<WizardData>({
    brandName: "",
    productImageBase64: null,
    purpose: "",
  });
  const [buildState, setBuildState] = useState<BuildState>("idle");

  const canAdvance = (): boolean => {
    switch (step) {
      case 1: return data.brandName.trim().length > 0;
      case 2: return data.productImageBase64 !== null;
      case 3: return data.purpose.trim().length > 0;
      case 4: return true;
      default: return false;
    }
  };

  // Navigation: back always allowed (except step 1), forward only if canAdvance
  const goBack = () => setStep((s) => Math.max(1, s - 1) as WizardStep);
  const goNext = () => {
    if (canAdvance()) setStep((s) => Math.min(4, s + 1) as WizardStep);
  };
  // ...render step components
};
```

### Pattern 2: Client-Side Image Resize
**What:** Read file via FileReader, load into Image element, draw onto canvas at reduced dimensions, export as base64.
**When to use:** When images need to be constrained to max dimensions before storage/transmission.
**Example:**
```typescript
// imageUtils.ts
const MAX_DIMENSION = 800;
const MAX_FILE_SIZE = 6 * 1024 * 1024; // 6MB

export function validateImageFile(file: File): string | null {
  if (!["image/png", "image/jpeg"].includes(file.type)) {
    return "Only PNG and JPG images are supported.";
  }
  if (file.size > MAX_FILE_SIZE) {
    return "Image must be under 6MB.";
  }
  return null; // valid
}

export function resizeImageToBase64(file: File): Promise<string> {
  return new Promise((resolve, reject) => {
    const reader = new FileReader();
    reader.onload = () => {
      const img = new Image();
      img.onload = () => {
        const { width, height } = img;
        let newWidth = width;
        let newHeight = height;

        if (width > MAX_DIMENSION || height > MAX_DIMENSION) {
          const ratio = Math.min(MAX_DIMENSION / width, MAX_DIMENSION / height);
          newWidth = Math.round(width * ratio);
          newHeight = Math.round(height * ratio);
        }

        const canvas = document.createElement("canvas");
        canvas.width = newWidth;
        canvas.height = newHeight;
        const ctx = canvas.getContext("2d")!;
        ctx.drawImage(img, 0, 0, newWidth, newHeight);

        // Use PNG for transparency support, JPEG for photos
        const mimeType = file.type === "image/png" ? "image/png" : "image/jpeg";
        const quality = mimeType === "image/jpeg" ? 0.85 : undefined;
        const base64 = canvas.toDataURL(mimeType, quality);

        // Strip data URL prefix for Office.js (it expects raw base64)
        const rawBase64 = base64.split(",")[1];
        resolve(rawBase64);
      };
      img.onerror = () => reject(new Error("Failed to load image"));
      img.src = reader.result as string;
    };
    reader.onerror = () => reject(new Error("Failed to read file"));
    reader.readAsDataURL(file);
  });
}
```

### Pattern 3: Image Region on Slide
**What:** Define a layout region for the product image and use ShapeFill.setImage() to fill a geometric shape.
**When to use:** When inserting an uploaded image onto a PowerPoint slide.
**Example:**
```typescript
// In slideRenderer.ts -- extend to handle product image
function addProductImage(
  shapes: PowerPoint.ShapeCollection,
  imageBase64: string,
  region: LayoutRegion
): void {
  const shape = shapes.addGeometricShape(
    PowerPoint.GeometricShapeType.rectangle,
    {
      left: region.left,
      top: region.top,
      width: region.width,
      height: region.height,
    }
  );
  shape.fill.setImage(imageBase64);
  shape.lineFormat.weight = 0; // No border
  shape.altTextDescription = "Product image";
}
```

### Anti-Patterns to Avoid
- **Global wizard state:** Do NOT use context/reducer for 4-step wizard state. Local useState in WizardPanel is sufficient and simpler.
- **Image in global state:** Do NOT store the base64 image anywhere beyond the WizardPanel component. It is session-only and should be garbage collected when the user switches tabs.
- **Sending image to Cube AI:** Do NOT send the base64 image to Cube AI in the prompt. The image is inserted directly on the slide via Office.js -- Cube AI only needs the brand name and purpose text.
- **Multiple PowerPoint.run() calls:** Do NOT create separate PowerPoint.run() calls for the image shape and content shapes. All shapes including the product image must be created within the single existing PowerPoint.run() in insertSlide().

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Image resize | Custom pixel manipulation | Canvas drawImage + toDataURL | Browser-native, handles all formats, aspect ratio math is trivial |
| File validation | Manual byte inspection | File.type + File.size checks | MIME type from browser is reliable for UI validation |
| Step navigation | Complex router-based wizard | Simple number state (1-4) with conditional rendering | 4 steps, no branching -- a switch statement is the right tool |
| Progress indicator | Custom SVG stepper | Fluent UI ProgressBar or simple styled divs | Visual-only, no interaction needed |

**Key insight:** This wizard is simple enough that adding any wizard library would be over-engineering. The entire state is: step number (1-4), brand string, image base64 or null, purpose string, and build status.

## Common Pitfalls

### Pitfall 1: ShapeFill.setImage Expects Raw Base64, Not Data URL
**What goes wrong:** Canvas.toDataURL returns `data:image/png;base64,iVBOR...` but ShapeFill.setImage() expects just `iVBOR...` (raw base64 without the data URL prefix).
**Why it happens:** Two different APIs with different expectations for the same data format.
**How to avoid:** Always strip the `data:...;base64,` prefix before passing to Office.js. Split on comma and take the second part.
**Warning signs:** Image shape appears but shows no image or shows an error placeholder.

### Pitfall 2: Canvas Loses Transparency When Using JPEG
**What goes wrong:** PNG images with transparency lose their alpha channel when resized via canvas.toDataURL("image/jpeg").
**Why it happens:** JPEG format does not support transparency.
**How to avoid:** Preserve the original MIME type through the resize pipeline. Use PNG output for PNG inputs, JPEG for JPEG inputs.
**Warning signs:** Product logos with transparent backgrounds get a white or black background.

### Pitfall 3: FileReader is Asynchronous
**What goes wrong:** Trying to read file.result before onload fires returns null.
**Why it happens:** FileReader.readAsDataURL is async but callback-based, not Promise-based.
**How to avoid:** Wrap FileReader in a Promise (as shown in Pattern 2 code example).
**Warning signs:** Image preview shows nothing, base64 is null or empty string.

### Pitfall 4: Large Base64 Strings in React State
**What goes wrong:** Re-renders become slow when a multi-MB base64 string is in component state.
**Why it happens:** React's reconciliation checks reference equality -- large strings are compared on every render.
**How to avoid:** The 800px max resize constraint (D-05) keeps the base64 small enough (typically under 200KB). If perf issues arise, store in a useRef instead of useState for the raw data, and keep only a boolean "hasImage" in state.
**Warning signs:** Noticeable lag when navigating between wizard steps after uploading an image.

### Pitfall 5: Tab Switching Unmounts the Wizard
**What goes wrong:** User uploads image in wizard, switches to "Ask Summit" tab, switches back -- all wizard state is lost.
**Why it happens:** Current App.tsx uses conditional rendering (`activeTab === "chat" ? <ChatPanel /> : ...`) which unmounts inactive tabs.
**How to avoid:** Either (a) keep all tabs mounted but hidden with `display: none`, or (b) lift wizard state to App.tsx and pass as props. Option (a) is simpler and preserves all component state.
**Warning signs:** User reports losing wizard progress when checking chat.

### Pitfall 6: Accept Attribute for File Input
**What goes wrong:** User selects a .gif or .webp file despite PNG/JPG restriction.
**Why it happens:** The `accept` attribute on `<input type="file">` is only a hint -- browsers allow override.
**How to avoid:** Use `accept="image/png,image/jpeg,.png,.jpg,.jpeg"` on the input AND validate `file.type` in the onChange handler. Show an error message for invalid types.
**Warning signs:** Non-PNG/JPG images are uploaded and cause issues downstream.

## Open Questions

1. **Image region positioning alongside data content**
   - What we know: Existing layouts (text-only, chart-text, table-text, full-combination) have defined regions. The product image needs a new region.
   - What's unclear: Should the image replace an existing region (e.g., the chart placeholder area) or should there be a new layout type (e.g., "image-text")?
   - Recommendation: Reuse the CHART region for the product image in chart-text and full-combination layouts. For text-only, add an image region on the right side (similar to chart-text TEXT/CHART split). This avoids creating an entirely new layout type.

2. **MIME type preservation through resize pipeline**
   - What we know: We need to store base64 for both display in the wizard (needs data URL prefix) and insertion via ShapeFill.setImage (needs raw base64).
   - What's unclear: Does ShapeFill.setImage() handle both PNG and JPEG base64 equally well?
   - Recommendation: Store raw base64 plus a separate mimeType string. Reconstruct data URL for `<img>` display. Pass raw base64 to ShapeFill.setImage(). Test both PNG and JPEG in manual verification.

3. **Review step content generation**
   - What we know: D-10 specifies showing title, layout type, bullet points, and image thumbnail.
   - What's unclear: The title and bullets come from Cube AI response -- does the review step call Cube AI BEFORE the user clicks "Build Slide", or does it just show the inputs (brand, purpose, image)?
   - Recommendation: The review step shows inputs only (brand name, purpose text, image thumbnail). The Cube AI call happens AFTER "Build Slide" is clicked. This avoids a wasted API call if the user goes back to edit.
