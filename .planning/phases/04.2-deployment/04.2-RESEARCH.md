# Phase 04.2: Deployment - Research

**Researched:** 2026-03-30
**Domain:** Docker + Railway deployment, webpack build-time env injection, Office Add-in manifest configuration
**Confidence:** HIGH

## Summary

This phase deploys the Summit VI add-in's production webpack build to Railway via Docker, replacing the current localhost:3100 development setup with a remotely-hosted HTTPS URL. The core work involves: (1) refactoring hardcoded secrets in `config.ts` to use webpack DefinePlugin build-time injection, (2) creating a multi-stage Dockerfile that builds the webpack bundle then serves it with nginx, (3) updating webpack.config.js `urlProd` to the Railway subdomain, (4) updating `.gitignore` for security, and (5) pushing to a private GitHub repo that Railway deploys from.

Railway provides automatic HTTPS on its `*.up.railway.app` subdomains, handles Docker builds natively when a Dockerfile is present, and injects a `PORT` environment variable that nginx must listen on. The webpack `CopyWebpackPlugin` already has dev/prod URL replacement logic -- only the `urlProd` value needs changing. The manifest `AppDomains` element must be updated to include the Railway domain.

**Primary recommendation:** Use a multi-stage Dockerfile (node:22-alpine build stage + nginx:alpine-slim serve stage) with webpack DefinePlugin for env var injection. Railway auto-detects the Dockerfile, provides HTTPS, and injects PORT. The nginx config must use `$PORT` via envsubst since Railway assigns dynamic ports.

### Pattern 1: Webpack DefinePlugin for Build-Time Secrets
**What:** Replace hardcoded values in `config.ts` with compile-time constants injected via DefinePlugin
**When to use:** When secrets need to vary between environments but the app is client-side only

Current `config.ts`:
```typescript
export const CUBEAI_CONFIG = {
  baseUrl: "https://ai.gcp-us-central1.cubecloud.dev/...",
  apiKey: "sk-...",
  internalId: "simon.scott@summitinsights.com",
  timeoutMs: 180000,
};
```

Refactored `config.ts`:
```typescript
// Values injected at build time via webpack DefinePlugin
// In development: falls back to defaults for local dev
// In production: Railway env vars are injected during Docker build
declare const __CUBEAI_API_KEY__: string;
declare const __CUBEAI_BASE_URL__: string;
declare const __CUBEAI_EXTERNAL_ID__: string;
declare const __CUBEAI_TIMEOUT_MS__: number;

export const CUBEAI_CONFIG = {
  baseUrl: __CUBEAI_BASE_URL__,
  apiKey: __CUBEAI_API_KEY__,
  internalId: __CUBEAI_EXTERNAL_ID__,
  timeoutMs: __CUBEAI_TIMEOUT_MS__,
};
```

webpack.config.js addition:
```javascript
const webpack = require("webpack");

// In plugins array:
new webpack.DefinePlugin({
  '__CUBEAI_API_KEY__': JSON.stringify(process.env.CUBEAI_API_KEY || ''),
  '__CUBEAI_BASE_URL__': JSON.stringify(process.env.CUBEAI_BASE_URL || 'https://ai.gcp-us-central1.cubecloud.dev/api/v1/public/summitinsights/agents/11/chat/stream-chat-state'),
  '__CUBEAI_EXTERNAL_ID__': JSON.stringify(process.env.CUBEAI_EXTERNAL_ID || ''),
  '__CUBEAI_TIMEOUT_MS__': JSON.stringify(Number(process.env.CUBEAI_TIMEOUT_MS) || 180000),
}),
```
**Source:** [webpack DefinePlugin docs](https://webpack.js.org/plugins/define-plugin/)

**Key detail:** Use double-underscore naming (`__VAR__`) rather than `process.env.VAR` to avoid conflicts with Node's actual process object. DefinePlugin does direct text replacement, so `JSON.stringify()` is mandatory for string values.

### Pattern 2: Multi-Stage Dockerfile
**What:** Build webpack in a Node stage, copy dist/ to nginx
**When to use:** Any static site deployment to minimize image size

```dockerfile
# Stage 1: Build
FROM node:22-alpine AS build

WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci --ignore-scripts

COPY . .

# Build-time args from Railway env vars
ARG CUBEAI_API_KEY
ARG CUBEAI_BASE_URL
ARG CUBEAI_EXTERNAL_ID
ARG CUBEAI_TIMEOUT_MS

# DefinePlugin reads these during webpack build
ENV CUBEAI_API_KEY=$CUBEAI_API_KEY
ENV CUBEAI_BASE_URL=$CUBEAI_BASE_URL
ENV CUBEAI_EXTERNAL_ID=$CUBEAI_EXTERNAL_ID
ENV CUBEAI_TIMEOUT_MS=$CUBEAI_TIMEOUT_MS

RUN npm run build

# Stage 2: Serve
FROM nginx:alpine-slim

COPY nginx.conf /etc/nginx/nginx.conf.template
COPY --from=build /app/dist /usr/share/nginx/html

# Railway injects PORT dynamically -- envsubst replaces $PORT in nginx.conf
CMD ["/bin/sh", "-c", "envsubst '$PORT' < /etc/nginx/nginx.conf.template > /etc/nginx/nginx.conf && nginx -g 'daemon off;'"]
```

**Critical detail:** Railway injects `PORT` as a runtime environment variable. The Dockerfile CMD must use shell form (not exec form) to expand `$PORT` via `envsubst`. The exec form `["nginx", "-g", "daemon off;"]` will NOT expand environment variables.

**Source:** [Railway Dockerfiles docs](https://docs.railway.com/builds/dockerfiles)

### Pattern 3: nginx.conf with Dynamic PORT
**What:** nginx config that listens on Railway's injected PORT

```nginx
events {
    worker_connections 1024;
}

http {
    include /etc/nginx/mime.types;
    default_type application/octet-stream;

    server {
        listen $PORT;
        root /usr/share/nginx/html;
        index taskpane.html;

        location / {
            try_files $uri $uri/ /taskpane.html;
        }

        # Cache static assets
        location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg)$ {
            expires 1y;
            add_header Cache-Control "public, immutable";
        }

        # Security headers
        add_header X-Content-Type-Options "nosniff";
        add_header X-Frame-Options "SAMEORIGIN";
    }
}
```

**Note:** `$PORT` is replaced by `envsubst` at container startup, not by nginx. The `try_files` fallback to `taskpane.html` ensures the SPA works if any sub-paths are hit.

### Pattern 4: Manifest AppDomains Update
**What:** Add Railway domain to `AppDomains` so Office trusts it

The `manifest.xml` `AppDomains` section must include the Railway URL. The existing `www.contoso.com` placeholder should be replaced:


**Source:** [Microsoft AppDomain docs](https://learn.microsoft.com/en-us/javascript/api/manifest/appdomain)

### Anti-Patterns to Avoid
- **Using `process.env.X` directly in DefinePlugin keys:** Overwrites the entire `process` object, breaking Node module compatibility during build. Use custom names like `__VAR__` instead.
- **Exec form CMD with env vars:** `CMD ["nginx", "-g", "daemon off;"]` does NOT expand `$PORT`. Must use shell form or `envsubst`.
- **Hardcoding PORT in nginx.conf:** Railway assigns dynamic ports. Never hardcode port 80 or 3100.
- **Forgetting ARG declarations:** Railway injects env vars into Docker builds, but they are NOT available inside the Dockerfile unless declared with `ARG`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| HTTPS/SSL certificates | Manual cert generation | Railway auto-HTTPS | Railway provisions and renews certs automatically on *.up.railway.app |
| Static file serving | Custom Node.js Express server | nginx in Docker | nginx handles static files with zero code, gzip, caching headers |
| ENV var templating | Custom shell scripts | envsubst (built into nginx image) | Standard Unix tool, already in the nginx Alpine image |
| CI/CD pipeline | GitHub Actions workflow | Railway auto-deploy from GitHub | Railway watches the repo and rebuilds on push |

## Common Pitfalls

### Pitfall 1: Railway PORT Must Be Dynamic
**What goes wrong:** Container starts but Railway health check fails, deployment crashes
**Why it happens:** Hardcoding `listen 80` or `listen 3100` in nginx.conf. Railway assigns a random port via the `PORT` env var.
**How to avoid:** Use `envsubst` to template `$PORT` into nginx.conf at container startup
**Warning signs:** "Application failed to respond" error in Railway dashboard

### Pitfall 2: Docker ARG vs ENV Confusion
**What goes wrong:** DefinePlugin receives empty strings for all env vars during `npm run build`
**Why it happens:** Railway injects variables as build args, but they must be explicitly declared with `ARG` in the Dockerfile and then set as `ENV` for the shell to pass them to webpack
**How to avoid:** For each Railway env var: `ARG VAR_NAME` then `ENV VAR_NAME=$VAR_NAME` before `RUN npm run build`
**Warning signs:** Config values are empty/undefined in the built bundle

### Pitfall 3: DefinePlugin Requires JSON.stringify for Strings
**What goes wrong:** Build succeeds but runtime throws `ReferenceError: sk is not defined`
**Why it happens:** DefinePlugin does literal text replacement. Without `JSON.stringify()`, a string value `sk-abc` becomes the identifier `sk-abc` (minus operator) in the output
**How to avoid:** Always wrap string values: `JSON.stringify(process.env.MY_VAR)`
**Warning signs:** Syntax errors or ReferenceErrors at runtime for config values

### Pitfall 4: .gitignore Must Block Secrets Before First Push
**What goes wrong:** API key committed to git history permanently
**Why it happens:** Adding to `.gitignore` after committing doesn't remove from history
**How to avoid:** Update `.gitignore` FIRST, verify with `git status`, then commit. Never push `config.ts` with the hardcoded key.
**Warning signs:** `config.ts` or `config.ts.txt` showing in `git status` as tracked

### Pitfall 5: Manifest URL Count Mismatch
**What goes wrong:** Some icon URLs still point to localhost in production
**Why it happens:** The webpack `CopyWebpackPlugin` transform replaces `urlDev` with `urlProd` using a regex. If any URLs don't match `https://localhost:3100/` exactly, they won't be replaced.
**How to avoid:** After build, verify the `dist/manifest.xml` has zero instances of `localhost`
**Warning signs:** Missing icons in production, broken taskpane in sideloaded add-in

### Pitfall 6: sharp Native Module in Docker
**What goes wrong:** `npm ci` fails in Docker with `sharp` build errors on Alpine Linux
**Why it happens:** `sharp` has native dependencies and the project includes it as a devDependency
**How to avoid:** Use `npm ci --ignore-scripts` (sharp is only used for image processing in dev, not needed for the production webpack build) OR ensure the build stage uses a compatible base image. Since `sharp` is a devDependency and the Dockerfile runs `npm ci` (which installs devDependencies for the build), you may need `--ignore-scripts` or add `libc6-compat` to the Alpine image.
**Warning signs:** Build fails with "Could not load the sharp module" during `npm ci`
